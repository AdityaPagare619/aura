Enterprise Software Review Board Report: AURA v4
1. Executive Summary
AURA v4 is a highly ambitious, privacy-first, on-device AI assistant designed for Android (via Termux). Built primarily in Rust, it employs a two-process architecture separating deterministic execution and state management (aura-daemon) from non-deterministic LLM reasoning (aura-neocortex). This "Brain/Body Separation" is a strong architectural foundation that strictly limits the LLM's operational scope, preventing arbitrary code execution and uncontrolled prompt injection.

However, the current implementation exhibits critical integration gaps. The codebase fails to compile due to recent IPC contract changes that were not synchronized across crates. More concerningly, the core ReAct loop is "open"—relying on simulated outcomes in production paths rather than real device observations. While the mathematical models for personality (OCEAN) and data classification are robust, the system's operational readiness is low. The project requires significant integration work, testing infrastructure improvements, and error-handling maturation before it can reliably serve end users.

2. System Architecture Assessment
Strengths:

Brain/Body Separation: The strict division of responsibilities (Daemon = Body/Execution, Neocortex = Brain/Reasoning) is an excellent security and reliability pattern.
Tiered Memory Architecture: The 4-tier memory system (Working, Episodic, Semantic, Archive) correctly maps cognitive psychology concepts to technical storage implementations (RAM, SQLite, HNSW, ZSTD).
Policy Enforcement: The PolicyGate acts as a crucial safety valve, enforcing a deny-by-default stance independent of LLM whims.
Findings:

[CRITICAL] Open ReAct Loop (Integration Failure)

Location: crates/aura-daemon/src/daemon_core/react.rs
Problem: The SemanticReAct loop uses simulate_action_result() in production paths when a policy context is not provided or when the engine is not fully wired. The IPC contract for ReActObservation is missing or incomplete.
Reasoning: An agentic loop cannot function if the LLM does not receive ground-truth feedback about its actions.
Risk Impact: The system will fail to complete multi-step tasks or will hallucinate success, rendering the core value proposition non-functional.
Recommended Improvement: Define the ReActObservation IPC message. Refactor react.rs to ensure every action dispatched to the Executor results in a real screen hash/summary capture, which is then sent back to the Neocortex.
[HIGH] Heuristic Routing Violation

Location: crates/aura-daemon/src/routing/classifier.rs and react.rs
Problem: The routing system attempts to classify tasks using string matching to decide between S1 (DGS) and S2 (SemanticReAct).
Reasoning: This violates the "LLM = Brain, Rust = Body" architectural tenet. Rust should not attempt semantic understanding of user intent.
Risk Impact: Fragile routing that fails on novel phrasing, leading to incorrect execution paths.
Recommended Improvement: Replace heuristic routing with a simple binary check: Does this exact flow exist in the ETG cache? If yes, execute DGS. If no, route to SemanticReAct.
3. Security Risk Report
Strengths:

Anti-Cloud Manifesto: Zero telemetry and strictly local LLM inference eliminate a massive class of data exfiltration risks.
4-Tier Data Vault: Content-aware heuristics automatically classify sensitive data (e.g., PAN, Credit Cards) to Tier 3, requiring biometric auth.
Findings:

[HIGH] Deterministic Test Keys & Missing Keystore Integration

Location: crates/aura-daemon/src/persistence/vault.rs
Problem: The Vault implementation lacks wiring to the Android Hardware Keystore. Tests use deterministic keys ([0xAA; 32]), and there is no clear path to derive the production encryption_key safely if biometrics fail.
Reasoning: Without hardware-backed keystore integration, the encryption keys must be held in memory or derived solely from the PIN, making the system vulnerable to brute-force or memory-dump attacks on rooted devices.
Risk Impact: Complete compromise of Tier 2 and Tier 3 data if the device is compromised.
Recommended Improvement: Implement Android Keystore integration via JNI to wrap the Argon2id derived keys. Ensure the Vault absolutely refuses to initialize without a secure key.
[MEDIUM] Incomplete Argon2id Migration

Location: crates/aura-daemon/src/persistence/vault.rs
Problem: Comments indicate a planned migration from a weak hash to Argon2id, but the fallback logic for the migration is not explicitly implemented in the provided code, risking user lockout or silent fallback to weak hashing.
Reasoning: Security migrations must be atomic and robust.
Risk Impact: User lockout upon upgrade.
Recommended Improvement: Implement explicit, one-time migration logic that detects legacy hashes, verifies them once, re-hashes with Argon2id, and deletes the old hash.
4. Performance Risk Report
Findings:

[HIGH] IPC Serialization Overhead

Location: crates/aura-types/src/ipc.rs (ContextPackage::estimated_size)
Problem: The system relies on a crude estimated_size() loop to trim conversation history to fit within a 64KB IPC limit.
Reasoning: Trimming history arbitrarily drops context. Furthermore, computing sizes via string concatenation in a tight loop is inefficient.
Risk Impact: Loss of critical conversational or screen context, leading to poor LLM decisions. CPU spikes during context building.
Recommended Improvement: Implement a semantic compression tier (summarizing older turns via a fast LLM pass) or use a more efficient zero-copy IPC mechanism (e.g., shared memory via memfd or Ashmem) to bypass the 64KB socket limit entirely.
[MEDIUM] Blocking Operations in Async Contexts

Location: crates/aura-daemon/src/persistence/vault.rs
Problem: Cryptographic operations (Argon2id hashing, AES-GCM encryption) are performed synchronously.
Reasoning: Running heavy CPU-bound tasks on the main Tokio executor thread can starve async tasks.
Risk Impact: UI jank, missed heartbeat events, and delayed IPC responses.
Recommended Improvement: Wrap all cryptographic operations in tokio::task::spawn_blocking.
5. DevOps & Production Readiness
Findings:

[CRITICAL] Broken Mainline Build

Location: Entire Workspace
Problem: cargo test --workspace fails due to tokens_used missing from enum instantiations and a UnixStream/TcpStream type mismatch in protocol.rs.
Reasoning: A broken mainline branch indicates a lack of pre-commit CI enforcement.
Risk Impact: Developer velocity grinds to a halt; regressions accumulate.
Recommended Improvement: Fix the compile errors immediately. Mandate GitHub Actions branch protection requiring cargo check and cargo test to pass on both host and aarch64 targets before merging.
[HIGH] Unreliable On-Device Compilation

Location: install.sh
Problem: The installation script builds Rust binaries natively on the Android device via Termux.
Reasoning: Mobile CPUs are not designed for sustained compilation workloads. This will cause severe thermal throttling, battery drain, and takes 10-30 minutes, leading to a terrible onboarding experience.
Risk Impact: High user abandonment rate during installation.
Recommended Improvement: Set up a CI pipeline to cross-compile the aura-daemon and aura-neocortex binaries and publish them as GitHub Releases. Have install.sh download the pre-compiled binaries by default.
6. AI System Assessment

Findings:

Reasoning: Accurate model geometry is critical for token budgeting.

Recommended Improvement: Fail fast if GGUF metadata cannot be read, or provide a strict, user-validated override mechanism in config.toml.
7. Code Quality Findings
Findings:

[HIGH] Pervasive Unsafe Error Handling

Location: Codebase-wide
Problem: The architecture document notes over 600 .unwrap() calls.
Reasoning: In a daemon process intended to run 24/7, panics are unacceptable.
Risk Impact: The daemon will crash unpredictably, requiring manual restarts via Termux and destroying user trust.
Recommended Improvement: Institute a strict "no-unwrap" policy enforced via clippy. Refactor critical paths using thiserror and ? propagation.

Location: crates/aura-daemon/src/ipc/protocol.rs

Reasoning: Clean OS abstractions are required for reliable cross-platform testing.
Recommended Improvement: Use cfg(target_os = "android") to cleanly separate Unix Domain Socket initialization from TCP fallback used for local desktop development.

1) Project Overview
AURA is a privacy-first, on-device AI assistant for Android devices designed to operate without any cloud dependencies. It targets users who demand absolute data sovereignty, high personalization, and persistent behavioral memory. The application is built using a two-process architecture in Rust: aura-daemon acts as the system's "body" (handling sensor data, persistent memory, screen interaction, and safety policies), while aura-neocortex serves as the "brain," managing LLM inference (powered by llama.cpp and quantized models like Qwen-3). Communication between these processes occurs over an IPC layer. The system features a 4-tier data classification vault, rule-based execution gates, and psychological heuristics (OCEAN and VAD models) for maintaining a cohesive agent persona. The app is deployed via Termux and cross-compiled for aarch64-linux-android.

Zero Cold-Start Failure Guarantee: The system is explicitly designed to function immediately upon initial boot with neutral baseline traits, degrading gracefully as it learns rather than failing without data.

Issue: routing/classifier.rs attempts heuristic string matching for routing, which violates the architecture rule that the LLM determines semantics. This should be simplified to a binary S1/S2 cache lookup.
