CRIT-CI-1 Toolchain Channel Split — nightly-2026-03-01 in repo vs stable in CI
Attribute	Detail
Files	rust-toolchain.toml, .github/workflows/ci.yml (lines 18, 34, 48, 62)

channel = "nightly-2026-03-01"
components = ["rustfmt", "clippy", "rust-src"]
targets = ["aarch64-linux-android"]
# ci.yml — remove dtolnay action, let rust-toolchain.toml govern
steps:
  - uses: actions/checkout@v4
  - run: rustup show  # reads rust-toolchain.toml automatically
  - run: cargo check --workspace
Verification: After fix, rustc --version in CI logs must show nightly-2026-03-01. Any other version is a regression.

CRIT-CI-2 release.yml Is Non-Functional — Missing Feature Flag and Submodule Fetch
Attribute	Detail
File	.github/workflows/release.yml (lines 12, 51)
Severity	Critical — release pipeline is broken
Description. The release workflow has two compounding failures:
1. Line 12 — Checkout without submodules. The actions/checkout@v4 step does not specify submodules: recursive. The aura-llama-sys crate wraps llama.cpp, which lives in a git submodule. Without recursive checkout, the llama-cpp/ directory is an empty pointer. The build cannot locate the C++ source.

CRIT-CI-3 install.sh — Placeholder SHA256 Checksums for Model Downloads
Attribute	Detail
File	install.sh (lines ~180–220)
Severity	Critical — supply-chain integrity voided
Cross-ref	CRIT-SEC-3 (§7 Security)
Description. The install script downloads ~4GB of quantized model files (Qwen-3 GGUF weights) and validates their integrity against SHA256 checksums. All checksums are placeholder strings:
EXPECTED_SHA256="PLACEHOLDER_REPLACE_BEFORE_RELEASE"
# ... repeated for each model variant
The validation logic exists and is structurally correct — it computes sha256sum of the downloaded file and compares against the expected value. But because the expected value is a literal placeholder, the comparison always fails, and a subsequent || true swallows the failure:
echo "$EXPECTED_SHA256  $model_path" | sha256sum -c - || true
Impact: An adversary performing a MITM attack on the model download (or compromising the hosting server) can substitute arbitrary model weights. The user receives a modified model with:
- Backdoored outputs (e.g., steering advice toward specific products/URLs)
- Altered safety guardrails (bypassing ethics rules)
- Data exfiltration encoded in model behavior
The || true means the user sees no warning. The install completes successfully with a compromised model.
Remediation:
# 1. Compute real checksums from canonical model files
QWEN3_4B_Q4_SHA256="a3f8b1c9d2e4f5a6b7c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9"
# 2. Hard-fail on mismatch — remove || true
echo "${QWEN3_4B_Q4_SHA256}  ${model_path}" | sha256sum -c - || {
    echo "FATAL: Model checksum mismatch. Download may be corrupted or tampered."
    echo "Expected: ${QWEN3_4B_Q4_SHA256}"
    echo "Got:      $(sha256sum "${model_path}" | cut -d' ' -f1)"
    rm -f "${model_path}"
    exit 1
}
Verification: Intentionally corrupt a downloaded model file (flip one byte). Install script must refuse to continue.

CRIT-CI-4 install.sh — Vault PIN Hashed with Unsalted SHA256
Attribute	Detail
File	install.sh (lines ~280–295)
Severity	Critical — authentication bypass
Cross-ref	CRIT-SEC-4 (§7 Security)
Description. During initial setup, the user is prompted for a vault PIN. The script hashes it with raw SHA256 and writes the digest to a config file:
pin_hash=$(echo -n "$user_pin" | sha256sum | cut -d' ' -f1)
echo "vault_pin_hash = \"$pin_hash\"" >> "$config_path"
Problems:
1. No salt. Identical PINs produce identical hashes. A precomputed rainbow table for 4–8 digit PINs contains ≤10⁸ entries (~3.2GB) — trivially storable.
2. No key stretching. SHA256 is a fast hash. On consumer hardware (RTX 4090): ~10 billion SHA256/sec. A 6-digit numeric PIN is brute-forced in <1ms. An 8-character alphanumeric PIN in <1 minute.
3. No pepper. The hash can be verified offline without access to any server-side secret.
Irony: The daemon's own vault implementation in vault.rs correctly uses Argon2id with salt, memory-hard parameters, and 4 parallel lanes. The install script bypasses the daemon's crypto and implements its own weaker version.
Remediation:
The install script should not implement its own crypto. Instead, it should delegate to the daemon:
# Option A: Delegate to daemon binary
"${install_dir}/aura-daemon" --init-vault --pin-from-stdin <<< "$user_pin"
# Option B: If daemon isn't available yet, use argon2 CLI
pin_hash=$(echo -n "$user_pin" | argon2 "$(head -c 16 /dev/urandom | xxd -p)" \
    -id -t 3 -m 16 -p 4 -l 32 -e)
Verification: Extract the stored hash. Confirm it is Argon2id format ($argon2id$v=19$m=...), not a raw 64-character hex string.

HIGH-CI-1 install.sh — Shell Injection via Unsanitized $user_name in sed
Attribute	Detail
File	install.sh (lines ~250–260)
Severity	High — arbitrary command execution
Description. The install script prompts the user for their name and injects it into a config file using sed:
read -rp "Enter your name: " user_name
sed -i "s/USER_NAME_PLACEHOLDER/$user_name/g" "$config_path"
The $user_name variable is expanded inside double quotes within a sed substitution. Special characters in sed replacement strings (/, &, \, newline) have semantic meaning:
User input	Effect
John/Jane	sed delimiter broken → syntax error
John&Doe	& means "matched text" in sed → corrupted output
John\nDoe	Newline injection → config file structure broken
x/g;e cat /etc/passwd;#	Arbitrary command execution via e flag (GNU sed)
Remediation:
# Option A: Escape sed special characters
escaped_name=$(printf '%s\n' "$user_name" | sed 's/[&/\]/\\&/g')
sed -i "s/USER_NAME_PLACEHOLDER/${escaped_name}/g" "$config_path"
# Option B (preferred): Avoid sed entirely
python3 -c "
import sys
path = sys.argv[1]
name = sys.argv[2]
content = open(path).read()
content = content.replace('USER_NAME_PLACEHOLDER', name)
open(path, 'w').write(content)
" "$config_path" "$user_name"

CRIT-TEST-1 integration_tests.rs — ~45 Hollow Tests with Tautological Assertions
Attribute	Detail
File	crates/aura-daemon/src/integration_tests.rs (~37KB)
Severity	Critical — false confidence in system integration
Description. The integration test file contains approximately 45 test functions. Despite the file's size (37KB), these tests perform no meaningful behavioral verification. They exhibit three systematic patterns of tautology:

- Memory allocation failures
- Incorrect default initialization of internal data structures

The test constructs a PolicyGate with DenyByDefault, then asserts the mode is DenyByDefault. fn test_working_memory_capacity_enforcement() {
    let mut wm = WorkingMemory::new(4).unwrap(); // Small capacity for testing
    // Fill to capacity
    for i in 0..4 {
        wm.store(format!("item_{i}"), MemoryEntry::text(format!("value_{i}")))
          .expect("should accept within capacity");
    }
    // Verify at capacity
    assert_eq!(wm.len(), 4);
    assert_eq!(wm.remaining(), 0);
    // Verify eviction on overflow — LRU item should be evicted
    wm.store("item_4".into(), MemoryEntry::text("value_4".into()))
      .expect("should evict LRU and accept");
    assert_eq!(wm.len(), 4); // Still at capacity
    assert!(wm.get("item_0").is_none(), "LRU item should be evicted");
    assert!(wm.get("item_4").is_some(), "new item should be present");
}
#[test]
fn test_working_memory_retrieval_updates_access_time() {
    let mut wm = WorkingMemory::new(3).unwrap();
    wm.store("a", entry("va")).unwrap();
    wm.store("b", entry("vb")).unwrap();
    wm.store("c", entry("vc")).unwrap();
    // Access "a" — moves it from LRU to MRU
    let _ = wm.get("a");
    // Insert "d" — should evict "b" (now LRU), not "a"
    wm.store("d", entry("vd")).unwrap();
    assert!(wm.get("a").is_some(), "recently accessed, should survive");
    assert!(wm.get("b").is_none(), "LRU after 'a' was accessed, should be evicted");
}

CRIT-CI-4 — Unsalted SHA256 PIN Hashing in install.sh
Field	Detail
File	install.sh (lines 245–260)
Severity	Critical — authentication mechanism is trivially defeated
Cross-ref	CRIT-SEC-4 (§7 Security Review)
Description. The installation script's vault PIN setup hashes the user-provided PIN with raw SHA256:
pin_hash=$(echo -n "$pin" | sha256sum | awk '{print $1}')
No salt. No key stretching. No iteration count. A 4–6 digit numeric PIN has at most 1,000,000 possible values. At SHA256 speeds (~10 billion hashes/second on consumer GPU), the entire keyspace is exhausted in <1 millisecond.
Note: The Rust daemon code correctly uses Argon2id for vault key derivation. This finding is isolated to the install.sh initial setup path. However, the PIN hash written by install.sh is stored and later read by the daemon — if the daemon trusts this hash for authentication, the weak link is exploitable.
Remediation (required before merge):
- Replace with Argon2id (matching the daemon's own implementation)
- Or delegate PIN setup to the daemon binary itself, avoiding shell-based crypto entirely

HIGH-CI-1 — Shell Injection via Unsanitized $user_name in sed
Field	Detail
File	install.sh (line ~310)
Severity	High — arbitrary command execution
sed -i "s/%%USERNAME%%/$user_name/g" "$config_path"
The variable $user_name is sourced from interactive user input with no sanitization. The sed s command uses / as its delimiter. If the user enters a name containing /, the sed expression breaks. If the user enters foo/e s/^/rm -rf \//e (or similar), sed executes arbitrary shell commands via the e flag on systems where GNU sed is present.
Attack surface: The install.sh script runs with user privileges, but on Android via Termux, this may include access to the full app sandbox and sdcard.
Remediation:

escaped_name=$(printf '%s\n' "$user_name" | sed 's/[&/\]/\\&/g')
sed -i "s/%%USERNAME%%/$escaped_name/g" "$config_path"
# Option B: Use awk (no delimiter collision)
awk -v name="$user_name" '{gsub(/%%USERNAME%%/, name)}1' "$config_path" > tmp && mv tmp "$config_path"

HIGH-CI-2 — NDK Downloaded Without Integrity Verification
Field	Detail
File	install.sh (lines ~95–120)
Severity	High — toolchain supply-chain risk

HIGH-TEST-2 — All Executor Tests Bypass PolicyGate via for_testing()
Field	Detail
File	crates/aura-daemon/src/execution/ (test modules)
Severity	High — security-critical path is never tested in context
Every executor test constructs the executor using:
let executor = Executor::for_testing(mock_tools);
The for_testing() constructor creates an AllowAllPolicyGate — a gate that approves every action without evaluation. The production constructor uses DenyByDefaultPolicyGate, which requires explicit allowlist matching.
Consequence: The PolicyGate module itself is well-tested in isolation (§10.7 shows ~80 strong tests). But the integration of PolicyGate with the Executor pipeline is never tested. Specifically:
- Does the Executor correctly call gate.evaluate() before executing?
- Does a Deny result prevent execution?
- Does the Executor propagate the denial reason to the caller?
- Does the audit log record denied actions?
None of these questions have test-backed answers.
Remediation:
#[tokio::test]
async fn test_executor_respects_production_policy_gate() {
    let gate = DenyByDefaultPolicyGate::with_rules(vec![
        AllowRule::tool("calculator"),
    ]);
    let executor = Executor::with_gate(gate, mock_tools);
    // Allowed action
    let result = executor.execute(Action::tool("calculator", "2+2")).await;
    assert!(result.is_ok());
    // Denied action
    let result = executor.execute(Action::tool("file_write", "/etc/passwd")).await;
    assert!(matches!(result, Err(ExecutorError::PolicyDenied(_))));
}

HIGH-DOC-1 — Two ReAct Loops with Different Limits, Undocumented
Field	Detail
Files	crates/aura-daemon/src/daemon_core/react.rs, crates/aura-neocortex/src/inference/react.rs

The daemon's ReAct engine uses MAX_ITERATIONS = 10. The neocortex's inference-level ReAct loop uses MAX_REACT_ITERATIONS = 5. These are two distinct loops operating at different abstraction levels:

Remediation: Add an architecture note documenting both loops, their relationship, and the worst-case multiplicative behavior.

### 11.8 Root Cause Analysis
The documentation inconsistencies follow a clear pattern:
1. **Documents were written aspirationally before code was finalized.** The Identity Ethics and Operational Flow documents describe a *planned* system with 4 trust tiers and 15 ethics rules. The code evolved to 5 tiers and 11 rules during implementation. The documents were never updated.
2. **No single source of truth is designated.** Five documents define trust tiers because no document is marked as canonical. The Ground Truth Architecture *should* be canonical (and is the most accurate), but it doesn't declare itself as such.
3. **No automated consistency checks exist.** There are no tests that assert documentation claims against code constants. A simple test like `assert_eq!(TrustTier::VARIANT_COUNT, 5)` with a comment linking to the docs would have caught CRIT-DOC-1 the moment the fifth tier was added.
4. **Documentation reviews don't happen alongside code reviews.** The git history shows code changes without corresponding doc updates. PRs modify `relationship.rs` without touching any markdown file.
