## All Findings
### 🔴 CRITICAL
**[FINDING-1] `build.rs` line 20 — `#[cfg(not(feature = "stub"))]` is always evaluated at build time on the host**
- **File:** `crates/aura-llama-sys/build.rs:20`
- **Description:** `build.rs` executes on the BUILD HOST, not the target. The `#[cfg(feature = "stub")]` attribute in build.rs is processed by the Rust compiler for the build script itself. **When `--features stub` is NOT passed** (i.e., in `build-android.yml`), the inner block tries to compile `llama.cpp/*.c` files. The `#[cfg(not(feature = "stub"))]` guard in a build script is **not a proc-macro** — it IS actually evaluated correctly because build scripts are compiled with the feature flags of their package. However, the confusion risk is high: if `build.rs` is ever refactored to use runtime `env::var("CARGO_FEATURE_STUB")` instead, the host-vs-target confusion (noted in the header comment) would resurface. The current implementation is marginally correct but fragile. The idiomatic approach is `if std::env::var("CARGO_FEATURE_STUB").is_ok() { return; }` at the top of `main()`, not scattered `#[cfg]` attributes in a build script.
**[FINDING-2] `release.yml` line 51 — `cargo check --workspace` missing `--features stub`**
- **File:** `.github/workflows/release.yml:51`
- **Severity:** CRITICAL — the CI gate in release.yml runs `cargo check --workspace` (no `--features stub`) but the checkout at line 31 has **no `submodules: recursive`**. This means: on a tag push, the release CI gate will try to `cargo check` `aura-neocortex` (which depends on `aura-llama-sys`), hit the build.rs, find that the target is `x86_64-linux` (not android), go to the `else` branch in build.rs and emit `llama_stub` — that part works. BUT the `cargo check` itself also compiles `aura-llama-sys` which links to `llama` (`links = "llama"` in Cargo.toml). Since no `llama_stub` linking is provided in `else` branch, the linker will fail looking for `libllama`. **Result: every single tagged release will fail its CI gate.**
  
  Correction — actually re-reading build.rs line 57: the else branch emits `cargo:rustc-cfg=llama_stub` and `cargo:stub=true` but does NOT emit `cargo:rustc-link-lib=static=llama`. Combined with `links = "llama"` in Cargo.toml, Cargo requires a `DEP_LLAMA_*` metadata or link directive when `links` is set. This may actually be fine IF the Rust source compiles without needing to link `libllama` when `llama_stub` cfg is set. Needs verification, but the missing `--features stub` flag is still a defect compared to ci.yml.
**[FINDING-3] `install.sh` lines 39/44/49 — ALL SHA256 checksums are placeholders**
- **File:** `install.sh:39`, `install.sh:44`, `install.sh:49`
- **Description:** `MODEL_QWEN3_8B_SHA256="PLACEHOLDER_UPDATE_AT_RELEASE_TIME_8b"` and equivalents for 4b and 14b. The `verify_checksum()` function at line 583 explicitly skips verification when the value starts with `PLACEHOLDER`. **A user installing AURA today downloads a multi-gigabyte model file with zero integrity verification.** This is a supply chain attack vector — a MITM or HuggingFace compromise would serve a poisoned model silently. The comment at line 581 confirms this is known (`# TODO: Set real checksums before release`) but it is in alpha release code already tagged `v4.0.0-alpha.1`.
**[FINDING-4] `install.sh` line 884 — PIN hashed with raw SHA256, no salt**
- **File:** `install.sh:884`
- **Description:** `pin_hash=$(echo -n "$pin1" | sha256sum | cut -d' ' -f1)` stores an **unsalted SHA256** of a 4-digit PIN to disk at `config.toml` as `pin_hash = "sha256:<hash>"`. There are only 10,000 possible 4-digit PINs. An attacker who reads the config file can brute-force all 10,000 possibilities in milliseconds with a rainbow table. The comment on line 882 says "Real bcrypt would be done by the daemon on first run" — but there is no guarantee the daemon actually upgrades it (no code path for that reviewed here). This is a false security promise.
**[FINDING-5] `rust-toolchain.toml` vs CI toolchain — silent override**
- **File:** `rust-toolchain.toml:2`, all workflows
- **Description:** `rust-toolchain.toml` pins `channel = "nightly-2026-03-01"` with `components = ["rustfmt", "clippy", "rust-src"]`. All three CI workflows use `dtolnay/rust-toolchain@stable`. When the action installs `stable` and then Cargo runs, **rustup reads `rust-toolchain.toml` and overrides to nightly**. The `with: components: clippy` in the workflow action is applied to the stable toolchain, but the build actually uses nightly. This means: (a) CI may succeed with nightly features that would break on stable, (b) MSRV cannot be enforced, (c) if nightly-2026-03-01 has a regression, CI silently starts failing for undiagnosed reasons. The CI workflows must use `dtolnay/rust-toolchain@nightly-2026-03-01` or simply remove the `rust-toolchain.toml` conflict.

FINDING-9 install.sh line 671-673 — JNI library logic is conceptually incorrect
- File: install.sh:670-674
- Description: The script copies a natively-compiled Termux binary (built without JNI exports, targeting the host Termux ABI for CLI use) to android/app/src/main/jniLibs/arm64-v8a/libaura_daemon.so. A Termux CLI binary is NOT the same as an Android JNI library — it has a main() entry point, not Java_* JNI exports, and is linked differently. Copying it as a .so file will cause System.loadLibrary() to crash at runtime with UnsatisfiedLinkError. The CI's build-android.yml correctly builds the cross-compiled binary separately. The install script's JNI step is vestigial/incorrect.
FINDING-10 install.sh line 459 — shallow clone loses submodule
- File: install.sh:459
- Description: git clone --depth 1 --branch "$target_ref" "$AURA_REPO" "$AURA_HOME" performs a shallow clone but does NOT initialize submodules (--recurse-submodules is absent). The aura-llama-sys/llama.cpp submodule will be empty after this clone. Since the build phase runs cargo build (NOT with --features stub), it will hit the panic in build.rs:22-28. The fix: git clone --depth 1 --recurse-submodules --shallow-submodules --branch "$target_ref".
FINDING-11 release.yml — third-party action not pinned to commit SHA
