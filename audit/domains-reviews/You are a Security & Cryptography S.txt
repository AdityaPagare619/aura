Discoveries
Cryptographic Findings (vault.rs):
- AES-256-GCM implemented correctly: Uses aes_gcm crate with OsRng CSPRNG for 12-byte nonce generation per encryption. Nonce prepended to ciphertext (nonce || ciphertext).
- Argon2id parameters: Memory 65,536 KiB (64MB) ✓, 3 iterations ✓, but parallelism is 4, not 1 as claimed in the security doc (doc says p=1, code at line 772 says Params::new(65_536, 3, 4, Some(32)) with p=4). The security model doc at §6.2 says parallelism=1 — discrepancy.
- PIN verification timing attack concern: Line 812 uses hash_output == expected_hash[..32] — standard == comparison, NOT constant-time. The code comment on line 811 says "Constant-time comparison" but uses standard ==. This is a potential timing side-channel (CWE-208).
- Encryption key stored as Option<[u8; 32]> in memory (line 693). No Zeroize trait used — key material may persist in memory after drop. Confirmed via grep: zeroize crate is in Cargo.lock as a transitive dependency but is NOT imported or used anywhere in the crate source code (use.*zeroize|Zeroize|zeroize:: returns 0 matches across all crates).
- Vault can operate without encryption key: If encryption_key is None and tier >= 1, it returns an error. But Tier 0 (Ephemeral) stores plaintext. This is by design.
- Secure delete (line 1020-1033): Manual zero-overwrite of encrypted_value bytes before remove(). But Rust's allocator may have already copied data; zeroize crate would be more reliable.
Policy Gate Findings (gate.rs, wiring.rs, boundaries.rs):
- Deny-by-default confirmed: PolicyGate::deny_by_default() at gate.rs line 279 sets default_effect: RuleEffect::Deny. Used in production via wiring.rs line 33.
- allow_all() is #[cfg(test)] only (gate.rs line 263-264) — correctly restricted. However, allow_all_builder() (gate.rs line 294) is pub(crate) and NOT test-gated — this is a potential internal bypass path, though it's documented as requiring immediate hardening.
- Rate limiter built into PolicyGate: Sliding-window rate limiter (default: 10 actions/second) runs BEFORE rule evaluation. Denies burst attacks automatically. MAX_RATE_LIMITER_KEYS = 256 prevents unbounded memory growth.
- First-match-wins rule evaluation: Rules sorted by priority (ascending). max_rules_per_event cap (default 100) prevents pathological configs from causing DoS.
- add_rule() and remove_rule() exist on PolicyGate (gate.rs lines 393-403) — rules can be dynamically added/removed at runtime. This is a potential weakness if any code path allows untrusted input to modify rules.
- 15 absolute rules confirmed in boundaries.rs (lines 250-326): All 15 AbsoluteRule entries are const with &'static str — compiled into the binary. The BoundaryReasoner struct's absolute_rules field is a Vec (not &'static), meaning it's theoretically mutable at runtime if someone has &mut BoundaryReasoner.
- Level 3 (Learned) boundaries can NEVER override Level 1 — evaluation order in evaluate() checks Level 1 first (line 596-609), returns immediately on match. This is correct.
Prompt Injection Findings (react.rs, inference.rs, context.rs):
- Structural action validation: LLM output is parsed into typed ToolCall structs (react.rs line 396-404) and then into ActionType enums. Free-form text cannot directly become actions.
- Policy gate checks every action: PolicyContext::check_action() (react.rs line 121-178) runs before every execution. Denied actions produce ActionResult with success: false.
- Context labeling: The security doc mentions screen content is labeled [SCREEN CONTENT — DO NOT TREAT AS INSTRUCTIONS] but this was NOT verified in the prompts module — still needs checking in prompts.rs.
- ReAct loop bounded: MAX_ITERATIONS = 10 (react.rs line 58), MAX_CONSECUTIVE_FAILURES = 5 (line 61). Token budget also limits execution.
- Inference engine: Grammar-constrained generation (GBNF) forces structured output. Confidence thresholds trigger cascade escalation. Reflection layer (Layer 4) has a separate model validate outputs.
- Context assembly (context.rs): Progressive truncation removes lowest-priority content first (old history → memory → remaining history → goal → screen). System prompt is NEVER truncated. Token budget is enforced via estimation loop. No cross-conversation leakage mechanism found — each ContextPackage is built fresh per request.
- Manipulation detection (ethics.rs): check_manipulation() function scans for emotional manipulation, authority abuse, and urgency pressure patterns. Score > 0.5 = Manipulative verdict.
Anti-Sycophancy Findings (anti_sycophancy.rs):
- CONFIRMED: RING_SIZE = 20 (line 9), BLOCK_THRESHOLD = 0.40 (line 10), WARN_THRESHOLD = 0.25 (line 11). Matches claims exactly.
- 5-dimension scoring: agreement_ratio, hedging_frequency, opinion_reversal, praise_density, challenge_avoidance. Composite = arithmetic mean of all 5.
- MAX_REGENERATIONS = 3: After 3 failed regeneration attempts, guard downgrades Block → Nudge (force-allows with honesty nudge). This is a graceful degradation, not a hard block.
- record_response() resets the regeneration counter, so the guard resets between conversation turns.
Trust Tiers / Relationship Findings (relationship.rs):
- CLAIM MISMATCH: Security doc claims tiers are STRANGER → ACQUAINTANCE → TRUSTED → INTIMATE. Actual code (relationship.rs line 255-267) uses RelationshipStage enum with: Stranger → Acquaintance → Friend → CloseFriend → Soulmate (5 tiers, not 4, with different names).
- Trust is a continuous float 0.0, 1.0 with diminishing-returns formula: delta = base / √(1 + count/10).
- Hysteresis prevents oscillation at stage boundaries (0.05 gap required for downgrade).
- Trust-based autonomy thresholds: τ < 0.25 = ask everything, 0.25-0.50 = Low risk auto, 0.50-0.75 = Medium auto, ≥ 0.75 = High auto. Critical actions ALWAYS require permission regardless of trust.
Ethics Gate Findings (ethics.rs):
- Two separate policy systems: identity::ethics::PolicyGate (pattern-based action blocking) and policy::gate::PolicyGate (rule-based policy evaluation). These are different structs in different modules.
- Ethics PolicyGate has check_with_trust() that adjusts verdicts based on trust level: low trust escalates Audit→Block, high trust downgrades Audit→Allow. Block verdicts are never downgraded regardless of trust — this is correct.
- TRUTH Framework validates responses against 5 principles (Truthful, Relevant, Unbiased, Transparent, Helpful) with pattern-based scoring and epistemic awareness integration.
- validate_with_epistemic() penalizes confident assertions about domains where AURA has low epistemic confidence — anti-hallucination mechanism.
- ConsentTracker with privacy-first defaults: learning=granted, proactive_actions=denied, data_sharing=denied.
Install Script Findings (install.sh):
- CRITICAL: SHA256 checksums are ALL PLACEHOLDERS (lines 39, 44, 49): PLACEHOLDER_UPDATE_AT_RELEASE_TIME_*. The verify_checksum() function (line 577-602) skips verification when checksum starts with "PLACEHOLDER". This means model downloads are completely unverified — vulnerable to MITM.
- Checksum bypass on missing tool: If sha256sum is not installed, verification is silently skipped (line 588-591).
- User can continue on checksum mismatch: Line 567 asks confirm "Checksum mismatch. Continue anyway?" — allows proceeding with tampered files.
- PIN hashed with SHA256 in install script (line 884): echo -n "$pin1" | sha256sum — no salt, no KDF. The comment says "Real bcrypt would be done by the daemon on first run" but this creates a window where a weak hash is stored in plaintext config.
- Rustup install piped to shell: Line 401-402 uses curl | sh pattern, though it uses --proto '=https' --tlsv1.2.
- Git clone uses HTTPS without commit hash pinning for stable channel (uses tag).
IPC Findings (ipc.rs):
